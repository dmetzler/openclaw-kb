# Feature Specification: Document Ingestion Pipeline with Semantic Chunking & Embeddings

**Feature Branch**: `009-doc-ingestion-pipeline`  
**Created**: 2026-04-15  
**Status**: Draft  
**Input**: User description: "Document ingestion pipeline with semantic chunking and vector embeddings for the OpenClaw Knowledge Base. Major enhancement to the ingestion pipeline: support for ANY document format (PDF, DOCX, PPTX, images) via docling, plus semantic chunking and embedding generation via Ollama."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Ingest a Local Document File (Priority: P1)

A knowledge base operator provides a file path to a local document (PDF, DOCX, PPTX, or image) to the ingestion pipeline. The system detects the document format, converts it to structured Markdown using docling, splits the content into semantically meaningful chunks that preserve document structure (sections, headings, tables), generates a vector embedding for each chunk via Ollama, extracts entities and relationships via LLM, creates or updates wiki pages and KG entities, and stores all chunks with their embeddings in the database. The operator sees progress feedback throughout the process.

**Why this priority**: Expanding ingestion beyond HTML/URLs to arbitrary document formats is the core value of this feature. Without it, the knowledge base cannot capture knowledge locked in PDFs, presentations, and Word documents — the most common knowledge containers in professional environments.

**Independent Test**: Can be fully tested by providing a multi-page PDF file, running the pipeline, and verifying that: the document is converted to Markdown preserving structure, content is split into semantically coherent chunks, each chunk has a 768-dimensional embedding stored in the database, wiki pages are created with correct frontmatter, KG entities exist in SQLite, and both chunks and embeddings are queryable.

**Acceptance Scenarios**:

1. **Given** a PDF file exists at a valid path, **When** the operator calls `ingestFile(filePath)`, **Then** the system converts it to structured Markdown via docling, preserving headings, paragraphs, tables, and page boundaries.
2. **Given** a DOCX file is provided, **When** the ingestion pipeline runs, **Then** the system produces the same quality output as PDF ingestion — structured Markdown with document hierarchy preserved.
3. **Given** a PPTX file is provided, **When** the ingestion pipeline runs, **Then** slide content is extracted and converted to Markdown with slide boundaries preserved as section markers.
4. **Given** an image file (PNG, JPEG) with text content is provided, **When** the ingestion pipeline runs, **Then** the system uses docling's OCR capabilities to extract text and convert it to Markdown.
5. **Given** the converted Markdown content, **When** the chunking step runs, **Then** the content is split into chunks of 200–500 tokens each, where each chunk preserves context metadata (parent section title, heading hierarchy, and page number for PDFs).
6. **Given** chunks have been created, **When** the embedding step runs, **Then** each chunk receives a 768-dimensional vector embedding generated by the Ollama nomic-embed-text model, stored in the database linked to its chunk record.
7. **Given** all chunks are embedded, **When** the entity extraction and page creation steps run, **Then** wiki pages and KG entities are created following the same flow as the existing URL ingestion pipeline.

### User Story 2 - Search by Semantic Similarity at Chunk Level (Priority: P1)

A user searches the knowledge base with a natural-language query. The system embeds the query text via Ollama, performs K-nearest-neighbor search against chunk embeddings, and returns the most relevant chunks with their parent entity context. This replaces the current entity-level vector search with finer-grained chunk-level search that surfaces the specific passages most relevant to the query.

**Why this priority**: Chunk-level semantic search is the primary consumer of the embeddings generated during ingestion. Without it, the embeddings have no user-facing value. Returning specific relevant passages rather than entire entity descriptions dramatically improves search precision and usefulness for LLM-powered question answering.

**Independent Test**: Can be fully tested by ingesting a document with distinct sections on different topics, searching for a topic covered in one specific section, and verifying that the returned chunks come from that section — not from unrelated sections of the same document.

**Acceptance Scenarios**:

1. **Given** chunks with embeddings exist in the database, **When** a user calls `searchSemantic("query text")`, **Then** the system embeds the query via Ollama, performs KNN search against chunk embeddings, and returns the most relevant chunks ranked by cosine similarity.
2. **Given** search results are chunk-level, **When** results are returned, **Then** each result includes the chunk text, its parent entity name and ID, the section heading hierarchy, and the similarity score.
3. **Given** the Ollama service is unavailable, **When** a semantic search is performed, **Then** the system falls back to FTS5 full-text search without errors, logging a warning about the degraded mode.
4. **Given** a query matches chunks from multiple entities, **When** results are returned, **Then** chunks from different entities are interleaved by relevance score rather than grouped by entity.

### User Story 3 - Enhanced URL Ingestion with Chunking and Embeddings (Priority: P1)

The existing `ingestUrl` function continues to work but now also chunks the extracted content and generates embeddings for each chunk. The existing readability-based fetch remains the conversion mechanism for HTML/URLs. The only change is that after content extraction, the content flows through the new chunking and embedding stages before entity extraction proceeds.

**Why this priority**: Existing URL ingestion must gain chunking and embedding capabilities to ensure all content in the knowledge base — regardless of source — is searchable at the chunk level. This is a non-breaking enhancement to the existing pipeline.

**Independent Test**: Can be fully tested by ingesting a URL, verifying that wiki pages and KG entities are created as before, and additionally verifying that chunks and chunk-level embeddings now exist for the ingested content.

**Acceptance Scenarios**:

1. **Given** a valid URL, **When** the operator calls `ingestUrl(url, llm)`, **Then** the pipeline produces all existing outputs (raw file, wiki pages, KG entities, index, log) plus new chunk records and embeddings.
2. **Given** the URL content is processed by readability (not docling), **When** the chunking step runs, **Then** the Markdown is split using heading-boundary chunking (h1/h2/h3 splits and paragraph boundaries) since docling's HierarchicalChunker is not applicable to readability output.
3. **Given** Ollama is not running, **When** the embedding step is reached, **Then** the pipeline completes successfully — chunks are stored without embeddings, and a warning is logged. The pipeline does not fail.

---

### User Story 4 - Backfill Embeddings for Existing Wiki Pages (Priority: P2)

A knowledge base operator runs a backfill script to process all existing wiki pages that do not yet have chunk-level embeddings. The script reads each Markdown file in `wiki/`, chunks it using heading-boundary chunking, generates embeddings for each chunk, and stores the chunks and embeddings in the database. Progress is reported throughout the process.

**Why this priority**: The existing knowledge base may contain hundreds of wiki pages ingested before this feature existed. Without backfill, those pages would be invisible to chunk-level semantic search, creating an inconsistent search experience.

**Independent Test**: Can be fully tested by creating several wiki pages without chunks/embeddings, running the backfill script, and verifying that all pages now have chunks and embeddings in the database, and that they appear in semantic search results.

**Acceptance Scenarios**:

1. **Given** wiki pages exist without corresponding chunk records, **When** the operator runs the backfill script, **Then** each page is chunked and embedded, and progress is reported (e.g., "Processing 5/42: wiki/entities/python.md").
2. **Given** some wiki pages already have chunks and embeddings, **When** the backfill script runs, **Then** already-processed pages are skipped without re-processing.
3. **Given** Ollama is not running when the backfill starts, **When** the script attempts to connect, **Then** it reports an error and exits cleanly without modifying any data.
4. **Given** a large wiki with many pages, **When** the backfill script runs, **Then** it processes pages sequentially with progress output, so the operator can monitor completion.

---

### User Story 5 - Full-Text Search Covers Chunk Content (Priority: P2)

When chunks are stored in the database, their text content is also indexed in the FTS5 search index. This means keyword-based full-text search now returns results from chunk content in addition to entity names and metadata, providing more comprehensive text search coverage.

**Why this priority**: FTS5 search is the fallback when Ollama is unavailable and the primary mechanism for exact keyword matches. Indexing chunk content ensures that detailed information within documents (specific facts, figures, names) is discoverable via keyword search even without vector embeddings.

**Independent Test**: Can be fully tested by ingesting a document containing a specific unique phrase buried in a paragraph, then performing an FTS5 search for that phrase, and verifying it appears in results with the chunk as the source.

**Acceptance Scenarios**:

1. **Given** chunks are stored in the database, **When** a chunk is inserted, **Then** its text content is automatically added to the FTS5 search index via a database trigger.
2. **Given** a chunk contains the phrase "mitochondrial membrane potential", **When** a user searches for "mitochondrial membrane", **Then** the chunk appears in FTS5 search results.
3. **Given** a chunk is deleted (e.g., when its parent entity is re-ingested), **When** the deletion occurs, **Then** the chunk's FTS5 index entry is also removed via cascade trigger.

---

### User Story 6 - Pass-Through for Plain Markdown and Text (Priority: P3)

When the ingestion pipeline receives a plain Markdown file or a text file (not PDF, DOCX, PPTX, or HTML), the content passes through without docling conversion. It is chunked using heading-boundary chunking and processed through the same embedding and entity extraction pipeline as any other content.

**Why this priority**: Markdown and text files are already in a usable format and do not need conversion. Supporting them ensures the pipeline handles all common document types without requiring the operator to pre-convert files.

**Independent Test**: Can be fully tested by providing a `.md` file, running `ingestFile`, and verifying it is chunked and processed without invoking docling.

**Acceptance Scenarios**:

1. **Given** a `.md` file is provided, **When** `ingestFile` runs, **Then** the file content is used directly without docling conversion, chunked by heading boundaries, and processed through the standard pipeline.
2. **Given** a `.txt` file is provided, **When** `ingestFile` runs, **Then** the file content is treated as plain text, chunked by paragraph boundaries, and processed through the standard pipeline.

---

### Edge Cases

- What happens when docling is not installed (Python package missing) and a PDF is provided?
- How does the system handle a PDF that docling cannot parse (corrupted, encrypted, password-protected)?
- What happens when Ollama is running but the nomic-embed-text model is not pulled?
- How does the system handle a document that produces zero chunks (empty or whitespace-only content)?
- What happens when the same document file is ingested twice — are chunks duplicated or replaced?
- How does the system handle very large documents (100+ pages) that produce thousands of chunks?
- What happens when Ollama returns an embedding with wrong dimensions (not 768)?
- How does the system handle a mix of text and non-text content in a single document (e.g., a PDF with mostly images)?
- What happens when chunk embedding partially fails (some chunks embedded, others not)?
- How does the backfill script handle wiki pages that have been manually edited since last ingestion?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide a document converter module that accepts any file path and detects the format based on file extension. For PDF, DOCX, PPTX, and image files (PNG, JPEG, TIFF), it MUST invoke docling via subprocess to convert to structured Markdown. For HTML/URLs, it MUST use the existing readability-based fetcher. For Markdown and plain text files, it MUST pass content through unchanged.
- **FR-002**: The document converter MUST call docling as a Python subprocess via Node.js `child_process`. Docling is installed as a Python package (`pip install docling`). The converter MUST handle subprocess failures (missing Python, missing docling package, conversion errors) by throwing descriptive errors without crashing the process.
- **FR-003**: System MUST provide a semantic chunker module that splits Markdown content into chunks of 200–500 tokens each. For content processed by docling, the chunker MUST use docling's HierarchicalChunker to split by document structure (sections, paragraphs, tables). For Markdown not processed by docling, the chunker MUST split on heading boundaries (h1/h2/h3) and paragraph boundaries.
- **FR-004**: Each chunk MUST be represented as an object containing: `text` (the chunk content), and `metadata` containing `source` (file path or URL), `section` (parent section heading hierarchy), `index` (chunk sequence number within the document), and `tokenCount` (approximate token count).
- **FR-005**: System MUST provide an embedder module that calls the Ollama API to generate 768-dimensional vector embeddings using the nomic-embed-text model. The module MUST export an `embed(text)` function returning a Float32Array of 768 dimensions, and an `embedBatch(texts)` function for efficient batch processing.
- **FR-006**: The embedder module MUST be configurable via environment variables: `OLLAMA_URL` (default: `http://localhost:11434`), `OLLAMA_MODEL` (default: `nomic-embed-text`), and `OLLAMA_EMBED_DIMENSIONS` (default: `768`).
- **FR-007**: When Ollama is unavailable (connection refused, timeout, service down), the embedder MUST NOT throw an error that halts ingestion. Instead, it MUST return `null` for the embedding and log a warning. The ingestion pipeline MUST continue without embeddings, storing chunks without vector data.
- **FR-008**: System MUST add a `chunks` table to the SQLite database via a new migration. The table MUST contain: `id` (primary key), `entity_id` (foreign key to entities with CASCADE delete), `chunk_index` (integer, position within entity), `content` (text), `metadata` (JSON), and `created_at` (timestamp).
- **FR-009**: System MUST add a `vec_chunks` virtual table using vec0 for chunk-level embeddings: `chunk_id` as integer primary key, `embedding` as float[768] with cosine distance metric.
- **FR-010**: The database abstraction layer (`db.mjs`) MUST be extended with functions: `insertChunk(entityId, chunkIndex, content, metadata)`, `getChunks(entityId)`, `deleteChunksForEntity(entityId)`, `upsertChunkEmbedding(chunkId, vector)`, and `findNearestChunks(queryVector, k)`.
- **FR-011**: System MUST provide a new `ingestFile(filePath, options)` entry point that orchestrates: format detection, document conversion, semantic chunking, entity extraction via LLM, wiki page and KG entity creation, chunk storage, and embedding generation for each chunk.
- **FR-012**: The existing `ingestUrl` function MUST be enhanced to include chunking and embedding stages after content extraction. The readability-based fetch mechanism MUST remain unchanged. Chunks MUST be stored and embedded following the same flow as `ingestFile`.
- **FR-013**: After creating wiki pages and KG entities, the ingestion pipeline MUST automatically chunk the content and generate embeddings for each chunk in a single pipeline execution. This embed-on-ingest behavior MUST be the default.
- **FR-014**: System MUST provide a backfill CLI script that processes all existing wiki pages lacking chunk records. The script MUST read each `.md` file in `wiki/`, chunk it using heading-boundary chunking, generate embeddings via Ollama, and store chunks and embeddings in the database. The script MUST report progress and MUST skip pages that already have chunks.
- **FR-015**: The semantic search function (`searchSemantic` in `wiki-search.mjs`) MUST be updated to embed the search query text via Ollama before performing KNN search. The search MUST operate on chunk-level embeddings (`vec_chunks`), returning the most relevant chunks with their parent entity context.
- **FR-016**: When a semantic search is performed and Ollama is unavailable, the search MUST fall back to FTS5-only results without throwing errors, consistent with the existing graceful degradation behavior.
- **FR-017**: Chunk content MUST be indexed in the FTS5 `search_index` table so that full-text keyword search covers chunk-level content. Triggers MUST auto-maintain the FTS5 index when chunks are inserted, updated, or deleted.
- **FR-018**: When a document is re-ingested (same file path or URL), existing chunks for the affected entity MUST be deleted and replaced with new chunks from the re-ingested content. Chunk replacement MUST be atomic (within a transaction).
- **FR-019**: The document converter MUST validate that the specified file exists and is readable before attempting conversion. For unsupported file types, the converter MUST throw a descriptive error listing supported formats.
- **FR-020**: The backfill script MUST validate that Ollama is reachable before starting processing. If Ollama is unavailable, the script MUST exit with a clear error message without modifying any data.

### Key Entities

- **Chunk**: A semantically meaningful segment of document content, linked to a parent entity. Contains the text content, positional metadata (index, section hierarchy), and optionally a vector embedding. Chunks are the atomic unit of semantic search.
- **Chunk Embedding**: A 768-dimensional vector representation of a chunk's text content, stored in the vec_chunks virtual table. Used for cosine-similarity KNN search at query time.
- **Document Conversion Result**: The structured Markdown output from converting a source document (PDF, DOCX, PPTX, image) via docling, preserving document hierarchy and structure.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An operator can ingest a PDF, DOCX, PPTX, or image file and have all resulting wiki pages, knowledge graph entities, chunks, embeddings, and search index updates completed in a single pipeline execution.
- **SC-002**: A semantic search query returns chunk-level results (specific passages) with parent document/entity context, enabling users to find relevant passages rather than entire documents.
- **SC-003**: All existing wiki pages can be backfilled with chunks and embeddings via the backfill script, with progress reporting and skip-already-processed behavior.
- **SC-004**: When the embedding service is unavailable, both ingestion and search continue to function using text-based mechanisms (chunking without embeddings, keyword search without vector search), with appropriate warnings logged.
- **SC-005**: Full-text keyword search returns results from chunk content, increasing the coverage of searchable content beyond entity names and metadata.
- **SC-006**: Re-ingesting the same document replaces old chunks atomically without leaving orphaned or duplicated chunk data.
- **SC-007**: The existing URL ingestion function continues to produce all its current outputs while additionally producing chunks and embeddings — no existing behavior is broken.
- **SC-008**: Plain Markdown and text files can be ingested without requiring the document converter, passing through directly to chunking and embedding.

## Assumptions

- Docling is installed as a Python package (`pip install docling`) and available on the system PATH. The Node.js wrapper calls it via `child_process.execFile`. If docling is not installed, only PDF/DOCX/PPTX/image ingestion fails — URL, Markdown, and text ingestion continue to work.
- Ollama is running locally at `http://localhost:11434` with the `nomic-embed-text` model pulled. Ollama runs on CPU on the N150 server at approximately 200–500ms per embedding, which is acceptable for batch ingestion. If Ollama is unavailable, the system degrades gracefully — chunks are stored without embeddings, and search falls back to FTS5.
- The existing SQLite database layer (spec 001), ingestion pipeline (spec 002), and hybrid search system (spec 005) are fully implemented and stable. This feature extends them without breaking existing functionality.
- The nomic-embed-text model produces 768-dimensional embeddings (its native output dimension). Both `vec_embeddings` and `vec_chunks` tables use 768 dimensions (`EMBEDDING_DIMENSIONS = 768` and `CHUNK_EMBEDDING_DIMENSIONS = 768`). The `OLLAMA_EMBED_DIMENSIONS` environment variable allows overriding the dimension if a different model is used.
- The existing `vec_embeddings` table (entity-level embeddings) remains for backward compatibility. The new `vec_chunks` table stores chunk-level embeddings. The semantic search is updated to query `vec_chunks` instead of `vec_embeddings`, but the old table is not removed.
- Docling's processing time on the N150 server is approximately 6 minutes for a 9-page PDF, which is acceptable for batch ingestion workflows. Real-time / interactive ingestion of large documents is not a goal.
- The LLM provider interface used for entity extraction (the `llm` parameter in `ingestUrl` and `ingestFile`) remains unchanged. Entity extraction operates on the full converted Markdown content, not on individual chunks.
- Chunk token counts are approximate, using a simple whitespace-based estimation (words / 0.75). Precise tokenizer-based counting is not required.
- The backfill script is a one-time or infrequent operation. It processes pages sequentially rather than in parallel, prioritizing simplicity and reliability over speed.
- The ingestion pipeline runs as a CLI command or programmatic function call — there is no web UI or API server in scope.
