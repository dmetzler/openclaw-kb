J'ai découvert le projet et la compagnie [Entire.io](https://www.linkedin.com/redir/redirect?url=http%3A%2F%2FEntire%2Eio&urlhash=bl90&trk=public_post-text). Créée par Thomas Dohmke (ex-CEO de GitHub), l'entreprise a levé 60 M$ en février dernier ; c'est beaucoup pour un outil. Mais pas n'importe quel outil. Je l'ai installé et je le teste sur mes projets Devoxx France. L'outil fonctionne avec Git. Il permet d'associer et de sauvegarder votre conversation avec un LLM, et le code qui a changé. Imaginez une pull-request où vous pouvez voir les fichiers modifiés par le LLM… et l'échange entre le développeur et l'agent de code (Codex CLI, Claude Code ou autre). Quel est l'intérêt ? Cela permet de faire des "revues de PR" beaucoup plus faciles. Vous voyez le code que votre collègue a modifié... et la discussion entre lui et le LLM. Vous voyez les tools utilisés, comment l'agent a écrit le code. Et vous avez aussi la possibilité d'optimiser/améliorer vos prompts... comme vous aviez la possibilité d'avoir des revues sur votre code. Je crois énormément en ce produit. C'est comme si j'avais découvert Github à son lancement. Il n'y a pas encore d'offre "entreprise", vous pouvez tester le client qui est open-source, et le SaaS est pour l'instant gratuit. Il faut un compte Github et il faut taper du code avec un clavier. Le site : [https://entire.io/](https://www.linkedin.com/redir/redirect?url=https%3A%2F%2Fentire%2Eio%2F&urlhash=qTVY&trk=public_post-text) Je ferai prochainement une vidéo de démo sur ma chaîne YouTube [https://lnkd.in/ecSmWmTJ](https://www.linkedin.com/redir/redirect?url=https%3A%2F%2Flnkd%2Ein%2FecSmWmTJ&urlhash=Weim&trk=public_post-text)

J'ai parcouru la doc et je ne suis pas sûr de comprendre l'intérêt de stocker toutes ces metadata sous forme de commits dans le repo lui-même (je comprends pour l'aspect co-localisation, mais au niveau performance j'ai du mal à voir la plu-value par rapport à une DB dans laquelle on aurait mis ces infos ... et puis ça me pose plein de questions : que se passe-t-il quand on rebase et qu'on modifie le sha1 de plein de commits ? il doit ré-indexer ?) J'attends de voir ta vidéo pour saisir ton engouement parce que ça me laisse un peu perplexe :-) Sur le fond, je suis d'accord avec toi sur la valeur que représente la discussion avec le LLM lors d'une code review, je ne le stockerais simplement pas au niveau du repo (vibe-kanban, par exemple, permet de partager les taches entre utilisateurs sur un projet ... la discussion avec le LLM reste par contre en local pour le moment, mais on pourrait imaginer qu'un jour cela soit partageable)
